# 科创综指盘中刷新、预估信号与持仓天数修复设计

## 目标

在不改变现有日线 MACD(12/18/13) 正式交易规则和历史回测口径的前提下：

1. A 股交易日的交易时段内每 30 分钟刷新最新净值、指数价格和 MACD 状态。
2. 盘中穿越只作为预警展示，并明确标注为“预估信号”。
3. 收盘后确认正式金叉或死叉信号，仍按下一交易日开盘执行。
4. 修复后端输出和 HTML 交易明细中的持仓天数全部为 0 的问题。

## 调度设计

GitHub Actions 使用 UTC cron，在周一至周五的以下北京时间触发：

- 上午：09:30、10:00、10:30、11:00、11:30
- 下午：13:00、13:30、14:00、14:30、15:00
- 收盘确认：15:30

交易所节假日不单独维护日历；数据源没有产生新交易数据时，脚本仍可运行，但只有 `signal_data.json` 内容实际变化时才提交。GitHub Actions 的计划任务可能因平台排队产生数分钟延迟，页面展示真实的生成时间和行情数据时间，避免将计划时间误认为实际更新时间。

## 信号口径

正式策略保持不变：当天 DIF 从小于等于 DEA 变为大于 DEA 时为日线金叉，反向穿越为日线死叉；正式信号使用收盘数据确认，并在下一交易日开盘执行。

盘中运行时，使用数据源返回的当日最新价格更新最后一根日线的暂估收盘价并重算 MACD：

- 暂估金叉：显示“预估买入信号”。
- 暂估死叉：显示“预估卖出信号”。
- 未发生穿越但 DIF/DEA 接近：继续显示现有逼近预警。
- 预估信号不写入正式历史信号或历史交易，不改变持仓和回测结果。

输出数据增加明确状态字段，包括信号方向、是否预估、行情时间、生成时间和展示文本。前端只根据这些字段展示，不自行推断正式或预估状态。

15:30 的收盘确认运行将已确认穿越标记为正式信号；正式历史信号和回测交易仍由完整日线序列生成。

## 净值口径

历史净值与回测指标继续使用正式日线数据。盘中页面额外展示使用最新指数价格估算的当前策略净值和基准净值，并标注为盘中估算；空仓状态下策略净值不随指数盘中价格变化。不得把盘中估算值写入月度、年度或历史回测序列。

## 持仓天数修复

根因是当前代码用 `hasattr(exit_date, 'days')` 判断 pandas `Timestamp`，条件恒为假，因此所有交易都写出 `0`。

后端将买卖日期统一转成 pandas 时间戳，并使用 `(exit_date - entry_date).days` 计算自然日持仓天数。输出保持整数 `holding_days`。HTML 使用该字段渲染“X 天”；字段异常或缺失时显示 `—`，不再静默显示 `0 天`。

这里采用自然日差，和表格中的买卖日期直观一致。例如 2026-06-18 到 2026-07-08 显示 20 天。

## 前端展示

页面当前信号区域增加醒目的状态标签：

- 盘中穿越：黄色/橙色“预估信号”标签，并显示“预估买入信号”或“预估卖出信号”。
- 收盘确认：沿用正式买入/卖出信号样式，并显示“收盘确认”。
- 盘中净值：数值旁显示“盘中估算”。

页面继续优先请求 `signal_data.json`，嵌入数据只作为请求失败时的后备。交易明细的持仓列使用修复后的后端字段并进行缺失值保护。

## 容错与数据新鲜度

- 数据源失败时保留现有缓存回退，但输出中标记数据来源和行情时间。
- 若行情日期或时间未前进，不伪造新的行情时间。
- 前端显示“行情时间”和“页面生成时间”，便于识别数据源延迟。
- 正式/预估状态由显式运行阶段和数据时点共同决定，避免本地时区与 GitHub UTC 环境造成误判。

## 测试与验收

先写失败测试，再实现代码。至少覆盖：

1. 跨月、跨年和同日交易的自然日持仓天数。
2. pandas `Timestamp` 输入不会再输出错误的 0。
3. 盘中金叉/死叉产生相应“预估信号”，且不进入正式历史交易。
4. 收盘确认信号保持正式日线口径。
5. 持仓与空仓时的盘中估算净值计算。
6. HTML 对正常持仓天数和缺失字段的展示。
7. 工作流 cron 覆盖约定的全部交易时段和 15:30 收盘确认时点。

验收时运行自动化测试、执行一次本地数据生成，并在浏览器中检查信号标签、更新时间、盘中净值标记和交易明细持仓天数。

## 非目标

- 不把策略改成 30 分钟 K 线 MACD。
- 不改变 MACD 参数、手续费或下一交易日开盘执行规则。
- 不为交易所节假日引入新的第三方日历服务。
- 不把盘中预估信号计入历史胜率、收益率或交易次数。
